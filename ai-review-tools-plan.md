# AI 审核工具规划

## 1. 目标

本页的目标不是提供一个完全自由的 AI Workflow 编排平台，而是为“题目审核”场景提供一组可配置、可复用、可稳定落库的 AI 审核工具。

核心原则：

1. 工具类型由系统预置，不允许用户无限自定义工具行为。
2. 每种工具的输出结构固定，便于前端展示、后端保存和后续统计。
3. 页面允许配置的内容主要是：模型、题目字段、提示词模板、是否启用。
4. 审核结论建议由 AI 给出，但最终提交结论仍建议由审核员确认。

## 2. 页面命名建议

推荐页面名称：`AI 审核策略`

说明：

1. `AI 设置` 负责维护 Provider、Endpoint、Model 路由。
2. `AI 审核策略` 负责维护审核场景下可执行的工具组合。
3. 一个策略可以包含多个工具步骤，用于辅助审核员完成题目审核。

## 3. 页面定位

该页面主要解决的问题：

1. 为审核员提供标准化的 AI 辅助审核能力。
2. 让不同题型、不同项目可以选择不同的审核策略。
3. 控制 AI 输出格式，避免因自由输出导致代码无法处理。

不建议在该页面支持的能力：

1. 任意定义步骤输入输出协议。
2. 任意定义后续自动动作。
3. 让 AI 返回完全不受约束的自然语言结果，并直接驱动系统逻辑。

## 4. 推荐的工具模型

建议采用“预置工具类型 + 可配置参数”的方式。

每个工具包含两层：

1. 工具类型：系统写死，定义用途、输入协议、输出协议。
2. 工具配置：页面可改，定义模型、字段、提示词模板、阈值等。

## 5. 可创建的审核工具

以下工具都适合放到“题目审核”场景中。

### 5.1 文本质量检查

用途：

1. 检查题目文本是否通顺。
2. 检查是否存在错别字、病句、歧义表达、格式错误。
3. 可用于题干、选项、答案、解析等任意勾选字段。

可配置项：

1. 使用模型
2. 参与检查的字段
3. 自定义提示词
4. 严格程度，例如宽松、标准、严格

建议固定输出：

```json
{
  "toolType": "TEXT_QUALITY_CHECK",
  "passed": true,
  "severity": "LOW",
  "summary": "整体表达通顺，存在个别标点问题",
  "issues": [
    {
      "field": "analysis",
      "type": "TYPO",
      "content": "推导过程存在错别字"
    }
  ],
  "suggestions": [
    "统一中文标点格式"
  ]
}
```

适用场景：

1. 所有题型通用
2. 作为审核策略中的基础步骤

### 5.2 AI 解题任务

用途：

1. 让 AI 根据题干、选项、附件等信息独立完成做题。
2. 产出 AI 的答案、解题思路、信心等级。

可配置项：

1. 使用模型
2. 题目字段选择，例如题干、选项、图片说明、材料内容
3. 是否要求输出详细解题过程
4. 题型提示，例如单选、多选、判断、主观题

建议固定输出：

```json
{
  "toolType": "AI_SOLVE_QUESTION",
  "answer": "B",
  "normalizedAnswer": "B",
  "reasoning": "根据题干条件可推导出正确选项为 B。",
  "confidence": 0.84,
  "rawConclusion": "选择 B"
}
```

说明：

1. 这是最核心的能力之一。
2. 提示词可以自定义，但返回字段必须固定。
3. 对于客观题，建议增加 `normalizedAnswer` 作为统一比对字段。

### 5.3 答案一致性比对

用途：

1. 比对标准答案和 AI 作答结果是否一致。
2. 判断是否“完全一致”“语义一致”“部分一致”“明显不一致”。
3. 适合客观题和部分主观题。

输入来源：

1. 标准答案
2. AI 解题任务的输出答案
3. 必要时补充题干与选项

可配置项：

1. 使用模型
2. 比对模式，例如严格字面一致、语义一致、步骤一致
3. 自定义提示词

建议固定输出：

```json
{
  "toolType": "ANSWER_MATCH_CHECK",
  "matchLevel": "EXACT",
  "isConsistent": true,
  "summary": "AI 作答与标准答案一致",
  "difference": null
}
```

建议枚举值：

1. `EXACT`
2. `SEMANTIC_MATCH`
3. `PARTIAL_MATCH`
4. `MISMATCH`
5. `UNKNOWN`

### 5.4 解题过程比对

用途：

1. 比对 AI 解题过程与标准解析是否一致。
2. 判断标准解析是否存在跳步、遗漏、逻辑错误或表达不足。
3. 用于辅助发现“答案对但解析差”的题目。

输入来源：

1. AI 解题过程
2. 标准解析
3. 标准答案
4. 题干和选项

可配置项：

1. 使用模型
2. 是否重点检查逻辑链路
3. 是否重点检查计算过程
4. 自定义提示词

建议固定输出：

```json
{
  "toolType": "REASONING_COMPARE",
  "isConsistent": true,
  "summary": "标准解析与 AI 推理主线一致，但标准解析略简略",
  "missingPoints": [
    "缺少中间推导步骤"
  ],
  "riskLevel": "MEDIUM"
}
```

### 5.5 难度评估

用途：

1. 根据题目内容、标准答案、标准解析、AI 解题过程综合评估题目难度。
2. 用于发现标注难度与实际难度不匹配的问题。

输入来源：

1. 题干
2. 选项
3. 标准答案
4. 标准解析
5. AI 解题过程

可配置项：

1. 使用模型
2. 难度分级体系，例如 3 级、5 级
3. 自定义提示词

建议固定输出：

```json
{
  "toolType": "DIFFICULTY_EVALUATION",
  "difficultyLevel": "MEDIUM",
  "score": 3,
  "summary": "需要两步推理，属于中等难度",
  "evidence": [
    "存在条件转换",
    "需要排除干扰项"
  ]
}
```

建议难度枚举：

1. `EASY`
2. `MEDIUM`
3. `HARD`

如果业务需要更细，可扩展为 5 级制，但前后端都必须统一。

### 5.6 题目完整性检查

用途：

1. 检查题目是否缺字段。
2. 检查题干、选项、答案、解析之间是否存在明显缺失或断裂。
3. 适合作为所有审核策略的预检查步骤。

可配置项：

1. 使用模型
2. 参与检查的字段
3. 自定义提示词

建议固定输出：

```json
{
  "toolType": "QUESTION_COMPLETENESS_CHECK",
  "passed": false,
  "summary": "解析字段过短，难以支撑审核",
  "missingFields": [],
  "warnings": [
    "解析内容不足"
  ]
}
```

### 5.7 审核总结建议

用途：

1. 汇总前面多个工具的结果。
2. 形成面向审核员的最终建议，不直接替代人工审核。
3. 给出推荐结论和风险说明。

输入来源：

1. 前置工具结果
2. 题目原始字段

可配置项：

1. 使用模型
2. 自定义总结提示词

建议固定输出：

```json
{
  "toolType": "REVIEW_SUMMARY",
  "recommendedDecision": "NEEDS_REVISION",
  "riskLevel": "MEDIUM",
  "summary": "答案基本正确，但解析完整性不足，建议退回修改",
  "keyIssues": [
    "解析步骤过少",
    "个别语句不够通顺"
  ]
}
```

建议结论枚举与当前审核系统保持一致：

1. `PASS`
2. `NEEDS_REVISION`
3. `REJECT`

## 6. 你当前提出的 4 个工具建议

你提到的方向是合理的，可以整理成以下标准工具：

1. `文本质量检查`
说明：对应“语句是否通顺，有错别字”，支持自由勾选字段。

2. `AI 解题任务`
说明：对应“提交 AI 做题，返回结果”，提示词可配，但返回结构必须固定。

3. `答案一致性比对`
说明：对应“标准答案和 AI 做题答案是否一致”。

4. `难度评估`
说明：对应“根据 AI 解题过程、标准解析、标准答案返回难度等级”。

建议补充两个基础工具：

1. `题目完整性检查`
2. `审核总结建议`

这样可以形成比较完整的一条审核链路。

## 7. 推荐的工具执行顺序

建议默认执行顺序如下：

1. 题目完整性检查
2. 文本质量检查
3. AI 解题任务
4. 答案一致性比对
5. 解题过程比对
6. 难度评估
7. 审核总结建议

说明：

1. 前两步用于做基础质量筛查。
2. 中间几步用于判断题目正确性、解析质量和难度合理性。
3. 最后一步专门生成审核建议，避免每个工具都直接输出审核结论。

## 8. 页面上建议允许配置的内容

每个策略建议允许配置：

1. 策略名称
2. 适用题型
3. 是否启用
4. 工具列表和顺序
5. 每个工具对应的模型
6. 每个工具读取哪些字段
7. 每个工具的提示词模板
8. 是否将某一步结果传给后续步骤

每个工具建议允许配置：

1. 工具名称
2. 工具类型
3. 使用模型
4. 字段映射
5. 提示词模板
6. 阈值参数，例如严格程度、置信度阈值、难度等级体系

## 9. 页面上不建议开放的内容

为了避免结果不可控，不建议页面开放以下能力：

1. 自定义工具输出字段
2. 自定义工具输出 JSON 结构
3. 自定义系统动作，例如自动通过、自动驳回、自动修改数据库
4. 自定义任意脚本执行逻辑

这些能力一旦开放，会导致：

1. 后端无法稳定解析结果
2. 前端无法稳定展示
3. 审核结果无法做统计和筛选
4. 同一个策略升级后会出现兼容问题

## 10. 动态验收标准的设计

不同项目、不同数据源、不同题型，确实可能存在不同的验收标准。

例如：

1. 某个项目要求 AI 连续作答 5 次。
2. 如果 5 次里有 3 次及以上答错，则说明这道题存在稳定性问题。
3. 这种判断不应该交给 AI 再做一次自由判断，而应该由系统规则来完成。

因此，页面设计不能只有“AI 工具”，还需要一层“规则判断”。

建议把策略中的步骤分成两类：

### 10.1 AI 工具步骤

负责调用模型，产出结构化事实结果。

例如：

1. 文本质量检查
2. AI 解题任务
3. 答案一致性比对
4. 解题过程比对
5. 难度评估
6. 审核总结建议

特点：

1. 需要调用模型
2. 返回固定 JSON 结构
3. 输出的是“分析结果”或“事实结果”
4. 不直接负责最终是否合格

### 10.2 规则判断步骤

负责根据前面步骤的结构化结果，做程序化判定。

例如：

1. 错误次数阈值判断
2. 正确率阈值判断
3. 多数投票判断
4. 综合条件判断
5. 分数区间判断

特点：

1. 不调用模型
2. 不依赖自由文本
3. 完全基于前置步骤的结构化字段计算
4. 输出稳定、可审计、可统计

## 11. 为什么“是否合格”不建议继续交给 AI

不建议新增一个 AI 工具专门判断“是否合格”。

原因：

1. AI 的结论天然存在波动性。
2. 同样的数据，多次调用可能得到不同判定。
3. 规则写在 prompt 中，不便于审计和排查。
4. 后续如果项目标准变化，修改 prompt 不如修改规则清晰。
5. 页面上虽然可以把它展示成一个步骤，但实现层应该是“规则步骤”，不是“AI 工具步骤”。

正确分工应该是：

1. AI 工具负责产出结果
2. 系统规则负责判断是否合格

## 12. 一个典型案例

需求示例：

1. 把题目交给 AI 连续作答 5 次。
2. 每次都和标准答案进行比对。
3. 如果 5 次里有 3 次及以上不一致，则判定这道题“命中风险规则”。

建议拆分为 3 个步骤：

### 12.1 步骤一：AI 解题任务

执行次数：5 次

固定输出示例：

```json
{
  "toolType": "AI_SOLVE_QUESTION",
  "runIndex": 1,
  "answer": "B",
  "normalizedAnswer": "B",
  "reasoning": "根据题干条件可推出 B",
  "confidence": 0.84
}
```

### 12.2 步骤二：答案一致性比对

每次解题结果都与标准答案比对一次。

固定输出示例：

```json
{
  "toolType": "ANSWER_MATCH_CHECK",
  "runIndex": 1,
  "standardAnswer": "C",
  "modelAnswer": "B",
  "matchLevel": "MISMATCH",
  "isConsistent": false
}
```

### 12.3 步骤三：规则判断

这一层不调用 AI，只根据前面的结构化结果计算。

规则配置示例：

```json
{
  "ruleType": "COUNT_THRESHOLD",
  "sourceToolType": "ANSWER_MATCH_CHECK",
  "metric": "isConsistent",
  "aggregate": "count_false",
  "operator": ">=",
  "value": 3,
  "sampleCount": 5,
  "result": "RISK_HIT"
}
```

规则输出示例：

```json
{
  "stepType": "RULE_EVALUATION",
  "ruleType": "COUNT_THRESHOLD",
  "result": "RISK_HIT",
  "passed": true,
  "summary": "5 次作答中有 3 次与标准答案不一致，命中稳定性风险规则",
  "metrics": {
    "totalRuns": 5,
    "mismatchCount": 3
  }
}
```

## 13. 推荐支持的规则类型

为了支持不同项目的不同验收标准，建议规则类型做成“有限模板”，不要做成任意脚本。

### 13.1 次数阈值规则

适合：

1. 错误次数 >= N
2. 命中问题数 >= N
3. 通过次数 < N

示例：

1. AI 作答 5 次，答错次数 >= 3，则判定风险较高

### 13.2 比例阈值规则

适合：

1. 正确率 >= 80%
2. 一致率 < 60%
3. 文本问题命中率 > 50%

示例：

1. 10 次作答中正确率低于 70%，则标记为不稳定

### 13.3 多数投票规则

适合：

1. 多次作答后取出现次数最多的答案
2. 多次难度评估后取主流难度等级

示例：

1. 5 次作答中有 4 次输出 B，则以 B 作为模型最终答案

### 13.4 组合条件规则

适合：

1. 同时满足多个条件才通过
2. 任一高风险条件命中则退回

示例：

1. 文本质量通过，并且答案一致性通过，才允许进入下一步

### 13.5 分数映射规则

适合：

1. 多个步骤各自产出分值
2. 汇总总分后映射为 PASS / NEEDS_REVISION / REJECT

示例：

1. 总分 >= 85 为通过，60-84 为退回修改，低于 60 为驳回

## 14. 页面设计建议更新

基于上面的分析，页面中的“可配置步骤”建议分为两类：

### 14.1 AI 工具步骤

页面允许配置：

1. 工具类型
2. 使用模型
3. 输入字段
4. 提示词模板
5. 执行次数
6. 超时、是否重试等参数

### 14.2 规则判断步骤

页面允许配置：

1. 规则类型
2. 依赖哪个前置步骤的输出
3. 聚合方式
4. 阈值
5. 命中规则后的结果标签
6. 对应的审核建议

说明：

1. 为了统一用户理解，页面上可以都展示为“步骤”。
2. 但实现层必须区分“AI 调用步骤”和“规则判断步骤”。
3. 前者调用模型，后者由代码计算。

## 15. 结论

本页面更适合设计成“AI 审核策略配置页”，而不是“完全自由工作流页面”。

建议采用以下边界：

1. 工具类型由代码预置。
2. 工具输入输出协议固定。
3. 用户只配置模型、字段、提示词、顺序、执行次数和开关。
4. 项目差异化标准通过“规则判断步骤”实现，而不是让 AI 自由决定是否合格。
5. AI 生成的是结构化审核建议，不直接代替人工最终审核。

按照这个设计，你提出的四类核心能力都可以落地，而且不会把系统做成一个不可控的通用工作流引擎。
